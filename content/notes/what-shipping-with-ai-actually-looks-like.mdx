I&rsquo;ve now shipped a few real, live websites using AI assistance.
Two on real domains for actual businesses:
[darksideautocare.com](https://darksideautocare.com) and
[miraandbloomevents.com](https://miraandbloomevents.com). A third,
this site, [zainbuilds.com](https://zainbuilds.com), is the most
deliberate of the bunch.

I want to write down what the workflow actually looks like before I
forget what was new about it.

## The shape of it

The pattern that kept working:

1. **Use ChatGPT to think before you build.** It has memory of how I
   talk, what I&rsquo;m working on, and what the project is for. I use
   it to shape the idea and engineer the prompts before any code is
   written.
2. **Use Claude to do the building.** Once the brief is clear, the
   prompts get handed to Claude. It&rsquo;s where the actual code,
   structure and longer pieces come from.
3. **VS Code, GitHub, Vercel.** Edit, version, deploy. In that order.
   Push early. Get a live URL on day one. Iterate against a real
   website, not an idea of a website.
4. **Read the code.** This is the one I keep being tempted to skip.
   Generated code I don&rsquo;t understand is technical debt before the
   site is even up. The whole point is being able to keep shaping it.

Four steps. Most of the work is in step 1.

## What it&rsquo;s good at

- Boilerplate. All of it. The time saved here is meaningful.
- Reading and explaining code I didn&rsquo;t write.
- Getting unstuck. The right prompt surfaces the issue faster than
  searching.
- The first version of anything. The first version is almost free now.

## What it&rsquo;s not

- A substitute for direction. You still have to know what the site is
  for.
- Always right. It will write confident, plausible code that does the
  wrong thing. You have to be the one checking.
- A taste machine. The default aesthetic is bland. Pulling the design
  toward something specific takes effort, every time.

## The unexpected part

The surprising part wasn&rsquo;t that AI could generate code &mdash;
that&rsquo;s the obvious part. It was that the whole process started to
become understandable.

I&rsquo;d heard words like deployment, production, prod, sandbox,
testing, workflow, approvals, errors, troubleshooting, pushing changes,
live updates &mdash; for years. They were ambient vocabulary, the way
&ldquo;the algorithm&rdquo; is ambient vocabulary. After actually doing
it &mdash; pushing a change, watching the build run, seeing the URL
update on my phone &mdash; the words stopped being walls. Each one had
a corresponding action and a screen.

That&rsquo;s the part that&rsquo;s changed what I think is possible.
Not the code. The fact that the process around the code is no longer
opaque.

## What I&rsquo;m doing differently with this site

This site is the most deliberate of the three because the brief was
written in full before any code was generated. A foundational strategy
document. Architecture decided on paper. Design system as design
tokens, not as a vibe. Content types defined before content was
written.

I&rsquo;ll know in six months whether the discipline of doing it that
way was worth it. The guess is yes.
